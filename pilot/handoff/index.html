2 · Airtable, the system of record

One base holds every person, event, and touch

All contacts and activity live here. Everything else reads from it. base: appQdixHbuttPldx6

  • contacts: one row per person. Key fields: events_signed_up, leader_ladder (Prospect → Core Leader), assigned_organizer, amendment5_commitments, and per-event status fields like signup_6_23_status / confirm_6_23_status / attendance_6_23_status.
  • contact_log: one row per touch (call, text, RSVP, attendance, commitment). This is where calls and RSVPs are recorded.
  • events, event_attendance, organizers: supporting tables.
The #1 data gotcha: RSVP counts must be counted as unique people, not log rows. contact_log has one row per action (plus the occasional duplicate or test entry), so counting rows inflates the number. The dashboards and the live spreadsheets all dedupe to unique people, which is why they are the trustworthy count.

3 · Backend, the Cloudflare Worker

A small API that does all the reading, writing, and cleanup

One serverless Worker (~40 endpoints) sits between everything and Airtable. It handles form submissions, the dashboards' data, magic-link logins, attendance, the commitment/conversion math, dedup, and the live CSV feed for the spreadsheets. repo: lizmckenna/groundwork → worker/

  • Login is a magic link to an allowlisted email (no passwords).
  • Deploy = npx wrangler deploy from the worker/ folder.
  • Secrets (Airtable key, admin/export keys) live in Cloudflare, not in the code.

5 · Turnout spreadsheets (Google Sheets)

Shared trackers for the regional leads, live-linked to Airtable

The leads track turnout in Google Sheets they already know. The trick: the "RSVPs (live)" tab pulls straight from Airtable via a Worker CSV feed and =IMPORTDATA(), so it's always the deduped hard count (no more hand-copying or numbers that disagree). Leads "claim" their people in a dropdown, which rolls up to each lead's goal.

  • One workbook per launch (Northland 6/18, KC 7/9). Tabs: Goals · RSVPs (live) · one per lead.
  • Refreshes about hourly; new RSVPs append to the bottom so claims stay aligned.
📈 Northland 6/18 tracker 📈 KC 7/9 tracker

7 · Your areas of responsibility, and what already helps

Area 1 — In-house Airtable capacity

  • Train regional co-leads and team leaders on Airtable for managing their base contact lists, commitment follow-ups, and event recruitment. The per-lead turnout trackers and the caller dashboards are the starting point.
  • List creation and curation for co-leads and team leaders to streamline the ideal flow of people. The live RSVP feed + claim sheets already do this per launch.
  • Tech support for Airtable issues, and systems to flag when follow-up and connections to the base aren't happening. The events dashboard surfaces a lot of this.

Area 2 — Google Calendar + Zoom infrastructure

  • Keep the PMOPS Google Calendars current with registrants from Airtable event tracking, especially in the 48 hours before an event.
  • Mass reminder emails to onboarding RSVPs 24h out, and follow-up emails to attendees and to no-shows within 24h. The dashboards expose the exact lists with copy-emails, and the Worker already sends Zoom confirmations.
  • Download each Zoom meeting's registrations and get attendance into Airtable within 18h. Partly built; the dashboard reads attendance the moment it's marked.

Optional / down the line: deeper data cleanliness (merging duplicates, normalizing the "who recruited" free text), and the September cutover from this Airtable + Worker stopgap to the full Groundwork app.
